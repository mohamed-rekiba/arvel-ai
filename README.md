# arvel-ai

An [arvel](https://pypi.org/project/arvel/) ecosystem package, generated by
`arvel new ai --package`.

```bash
uv sync
uv run pytest        # green out of the box, before you change anything
```

## How this skeleton works

Everything a package can contribute to a host app is **present but inert**: the
working pieces are small examples, the rest are commented verbs in
`provider.py`. Build by replacing the examples; prune by deleting what you
don't need. An un-pruned copy always boots and passes its own CI.

| Keep always | Why |
|---|---|
| `pyproject.toml` | entry point (auto-registration) + extras + tooling config |
| `src/arvel_ai/provider.py` | the integration point with the host app |
| `src/arvel_ai/config.py` | package defaults (host app values win) |
| `tests/` | the booted-app fixture is how packages test without an app |
| CI (`.github/workflows/ci.yml`) | ruff · mypy strict · import-linter · pytest |

| Delete freely | When |
|---|---|
| `manager.py` + `drivers.py` | no config-swappable backend (keep the fake if you keep the manager) |
| `facade.py` | no static-access sugar wanted |
| `routes.py` | package serves no HTTP |
| `commands.py` | no CLI contribution |
| `contracts.py` | no public Protocol to expose |
| `.github/workflows/release.yml` | not publishing to PyPI |

## Rules that keep a package healthy

- **Depend on arvel's shape, not its guts** — import from `arvel.contracts` and
  the public API only.
- **Heavy imports live inside driver methods**, never at module top; each
  engine gets an extra in `pyproject.toml`. The import-linter contract enforces it.
- **The fake driver is a first-class driver** — apps test your package the same
  way they test mail: swap the driver, assert on calls.

Full guide: the *Packages → Getting Started* page in the arvel docs.
